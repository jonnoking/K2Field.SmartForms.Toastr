ABOUT THE CUSTOM SMARTFORMS CONTROL PROJECT TYPE

The Custom SmartForms Control Project template is used in conjunction with the K2 Custom SmartForms Control Item templates to extend the standard set of K2 smartforms Controls. (SmartForm controls are essentially user interface components that designers can drag onto SmartForm Forms and Views. They are similar to controls in Visual Studio's toolbox, but they are specifically written for the K2 smartforms design tools.)

After creating a new Custom SmartForms Control project with the project template, choose from the available SmartForms Control Items templates and add to the project. Modify the item to suit your requirements. You can add multiple custom SmartForms Control items to the same project.

The following control templates are available:

SmartForms Basic Server Control Item

This is a basic control with server-side code only. This control does not include any styling, client-side scripting or style sheet.
Use this control template when you only need to write server-side code and do not need to raise events or handle client-side scripting with the control, or is you want to have full control over the implementation of the client-side interaction.  

SmartForms Basic Client  Control Item

This is a basic client-side control with server-side code, a client-side script file and stylesheet. The control includes properties that can be set at design time and at runtime with Rules.
Use this control template to start building a control that will implement basic client-side interaction.  

Note: the current implementation of this control does not support design-time styling, but runtime styling will work.

SmartForms Ajax Control Item

This is a control with Ajax support. The control exposes an event to execute the Ajax method and raises an event when the Ajax method completes.

 

Note:

SmartForms Controls can only be used on K2 smartforms.

These templates are only compatible with K2 smartforms 1.0.2 or later, since they utilize the smartforms Custom Control SDK.

To learn more about Custom SmartForms Controls, please refer to the following topic in the K2 SmartForms Developer Reference: K2 smartforms Custom Controls

You can also refer to the following K2 Underground group for samples of custom SmartForm Controls http://www.k2underground.com/groups/default.aspx?GroupID=66

DEVELOPMENT TASKS

To implement this Custom SmartForms Control template, you need to complete the following tasks:

1.      The target .NET Framework for the project should be set to .NET 3.5 (This should be set correctly by the project template)

2.      You must sign the assembly with a strong-name key. The project template includes a strong name key file already, but you may replace this key file with your own one, if you prefer.

3.      Use the Add New Item context menu on the project to add a one of the available SmartForms Control Item templates from the K2 Extensions category to the project

4.      The new item will add a selection of files to the project. The specific files will depend on the template you selected.
Note: some files (such as the definitions XML file, client-side script file, CSS files and so on) will automatically set to build as embedded resources. This is necessary since K2 smartforms uses reflection to extract resources from controls at design-time and runtime.

5.      Implement your custom control

a.      Define the Properties and Methods in the Control Type Definition XML file

b.      Implement client-side script code in the .js file

c.      Implement server-side script code in the .cs file

d.      Implement style sheet in the .css file

DEPLOYING AND INSTALLING CUSTOM SMARTFORM CONTROLS

Depending on your K2 environment, you may need to copy the resulting .dll to several machines. The custom control assembly (.dll) must be copied to the K2 smartforms web sites (design-time and runtime sites) and registered with the K2 server. Once the assembly is copied, you need to register the assembly with K2 smartforms. Refer to the sections below for more details.

K2 smartforms Designer Site
(in other words, all servers where the K2 Designer web site was installed)

Copy the assembly .dll file and any dependencies to the K2 Designer website's Bin directory.
Note: The K2 Designer site's Bin folder path is usually: "C:\Program Files (x86)\K2 blackpearl\K2 SmartForms Designer\Bin"

K2 smartforms Runtime Site
(in other words, all servers where the K2 Runtime web site was installed))

Copy the assembly .dll file and any dependencies to the K2 Runtime website's Bin directory.
Note: The K2 Runtime site's Bin folder path is usually: " C:\Program Files (x86)\K2 blackpearl\K2 SmartForms Runtime\Bin"

You should also run an IISReset for the web servers after the assembly is copied.

Registering the custom control assembly

To register the assembly with the K2 smartforms environment, use the utility controlutil.exe, located by default in [Program Files]\K2 blackpearl\bin\controlutil.exe. Run the command with the relevant switch and target the new assembly file. Examples below:

 

Register/update assembly: 
controlutil register -assembly:{assemblyfilepath}

De-Register/remove assembly: 
controlutil deregister -control:{controltypename}
(use this approach if you have multiple controls in the same assembly but only want to deregister one specific control)
controlutil deregister -assembly:{assemblyfilepath}
(use this command to deregister all the custom controls in an assembly)

List installed assemblies:
controlutil list

 

Note:
It is not always necessary to unregister the assembly before re-registering it. If you did not make any changes to the control's Definitions.xml file, it is not necessary to register the assembly again. If you have removed any controls from the assembly or have made any changes to the definition.xml file, you must re-register the assembly.

CUSTOM BUILD CONFIGURATIONS

To assist with developing the custom SmartForms Control project, the following custom build configurations are available in this project.

Note:
The built-in custom build configurations are only set up for scenarios where your development environment and K2 environment are on the same machine (for example, when you develop using one of the standard K2 virtual machines or if you have a local development server). You may need to edit the pre- and post-build commands to copy the output files to the correct location for your environment, or create your own build configurations.

 

Note:
The project will also output two batch files you can use to register or de-register the assembly.

DebugRegisterCopyLocal

Use this script whenever you change anything in the control definition .xml file, or the very first time you want to install the control.

 

The post-build script performs the following actions:

a)      Restarts IIS

b)      Copies the assembly file to the K2 Designer site at  C:\Program Files (x86)\K2 blackpearl\K2 SmartForms Designer\Bin

c)      Copies the assembly file to the K2 smartforms Runtime site at C:\Program Files (x86)\K2 blackpearl\K2 SmartForms Runtime\Bin

d)      Calls the controlutil.exe utility to register/update the assembly

 

DebugCopyLocal

Use this script when you have not made any changes to the control definition xml file. If you have only made changes to the control .cs, .resx, .js and .css files, you can use this build configuration.

 

The post-build script performs the following actions:

a)      Copies the assembly file to the K2 Designer site at C:\Program Files (x86)\K2 blackpearl\K2 SmartForms Designer\Bin

b)      Copies the assembly file to the K2 smartforms Runtime site at C:\Program Files (x86)\K2 blackpearl\K2 SmartForms Runtime\Bin

 

Note:
Remember to clear your browser cache whenever you change the control, since the cache may still have the old resources cached locally.

 

DEBUGGING THE CUSTOM SMARTFORM CONTROL

When debugging a custom smartform control, determine whether you are trying to debug the control at runtime (in other words, when it is being used on a SmartForm) or at design time (in other words, when you are designing a View and want to drag the control onto the View or otherwise configure the Control).

 

Design-time debugging

a)     Add breakpoints to your code

b)     To debug server-side code, attach to the w3wp process for the K2 Designer web site.

c)     To debug client-side code, use developer tools in IE or FireBug for Firefox to debug scripting, html and style sheet issues.

 

Runtime debugging

a)     Add breakpoints to your code

b)     Attach to the w3wp process for the K2 smartforms Runtime web site

c)     To debug client-side code, use developer tools in IE or FireBug for Firefox to debug scripting, html and style sheet issues.

 

KNOWN ISSUES

 

1.   Clear browser cache
Remember to clear your browser cache when testing custom controls. We recommend clearing your cache every time you update the custom control assembly. Unfortunately there is no reliable and consistent way to do this from the command line.

 

2.   Design-time styling
The Basic Client Control does not implement design-time styling or Preview styling. Runtime styling does work, however

 

3.   Nested namespaces
At the very top of the .js file for client controls, the namespace for the control is initialized. By default, the template's .js file supports a single-period namespace (e.g. Company.Control). If you use a nested namespace (for example, Company.Category.Control), you will need to add lines to the .js files for the client-side controls. Example below:
if (typeof Company === "undefined" || Company == null) Company = {};
if (typeof Company.Category === "undefined" || Company.Category == null) Company.Category = {};

 

4.   .SFC prefix for CSS class
To prevent conflicts with built-in controls, the CSS class locator for custom controls created with the SDK will be prepended with SFC.
